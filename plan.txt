Project Goal

Build a fault-tolerant asynchronous job scheduling system capable of:

Accepting jobs from multiple users
Processing jobs asynchronously
Tracking job state transitions
Recovering from worker failures
Retrying failed jobs
Maintaining complete execution history
Preventing job loss
Demonstrating distributed-system fundamentals

The goal is not maximum throughput.

The goal is:

Reliability
Consistency
Recoverability
Observability

Core Business Flow
Step 1

User submits a job.

Example:

Generate Report
Send Email
Process File

API receives request.

Step 2

System creates a job record.

Job State:

QUEUED

Stored in PostgreSQL.

Step 3

Job ID pushed into Redis queue.

Redis becomes temporary transport.

PostgreSQL remains source of truth.

Step 4

Worker pulls job from queue.

Worker ownership begins.

Job State:

RUNNING
Step 5

Worker periodically sends heartbeat.

Purpose:

I am alive.
I still own this job.

Heartbeat stored in database.

Step 6

Worker finishes.

Possible outcomes:

Success
RUNNING
    ↓
SUCCESS
Failure
RUNNING
    ↓
FAILED

or

RUNNING
    ↓
RETRY
    ↓
RUNNING

Security Model
Rule 1

Never trust URL parameters.

Bad:

/jobs/123

User changes:

/jobs/124

System returns another user's job.

Security failure.

Rule 2

Every job belongs to a user.

Validation:

Job ID
AND
User ID

must both match.

Rule 3

Use UUID everywhere.

Entities:

user_id
job_id
worker_id
execution_id
event_id

Reason:

Harder to enumerate
Globally unique
Future distributed compatibility
Rule 4

Store passwords hashed.

Never store:

Plain text passwords
Rule 5

JWT authentication.

Every request must identify:

Current user
Major Edge Cases
Worker Crash

Scenario:

Worker receives job

Machine crashes.

Result:

Job stuck forever

Solution:

Heartbeat Monitoring
Zombie Worker

Scenario:

Worker dies

Job remains RUNNING

Solution:

Watchdog Service

Detect stale heartbeat

Requeue job
Duplicate Execution

Scenario:

Worker finishes work

Crash before database update

Another worker retries.

Job runs twice.

Solution:

Idempotency

Every job must safely handle duplicate execution.

Database Failure

Scenario:

Worker completed task

Database unavailable

Solution:

Retry database update
Redis Failure

Scenario:

Queue unavailable

Effect:

No new jobs processed

Jobs remain safe because:

PostgreSQL is source of truth
Retry Storm

Scenario:

1000 jobs fail

All retry instantly

System overloads itself.

Solution:

Exponential Backoff

Example:

Retry 1 → 5s

Retry 2 → 30s

Retry 3 → 120s
Success Criteria

The project is considered successful if:

1. Jobs are never silently lost.

2. Worker crashes are recoverable.

3. Failed jobs can retry.

4. Job ownership is maintained.

5. Complete execution history exists.

6. Duplicate execution is handled.

7. Dashboard reflects real state.

8. Users cannot access other users' jobs.

9. Entire system runs locally.

10. Entire system runs through Docker Compose.
Final Mental Model

Do not think:

FastAPI Project

Think:

Reliable Job Lifecycle Manager

Everything in the system revolves around a single object:

JOB

The entire architecture exists to answer four questions:

Where is the job?

Who owns the job?

Who is processing the job?

What happens if something fails?
